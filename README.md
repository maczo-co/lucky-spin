# maczo — Lucky Spin Provably-Fair Verifier

Independently verify that every **Lucky Spin** result on [maczo.co](https://maczo.co) was fair and
was **not** altered after you played. This repo contains the exact algorithm the server uses, in a form
anyone can run **offline** — a single HTML page and a zero-dependency Node.js script — plus test vectors.

> This is the same style of open verifier that Stake and BC.Game publish: the game's randomness formula is
> fully public, so you don't have to *trust* the casino — you can *check* the math yourself.

---

## The fairness model (commit-first)

Each spin is decided by three values combined with cryptography:

| Value | Who controls it | When you see it |
| --- | --- | --- |
| **Server seed** | maczo (secret) | its **SHA-256 hash** is shown **before** you spin; the raw seed is revealed when you "Rotate" |
| **Client seed** | **you** (editable on the Lucky Spin page) | always |
| **Nonce** | server counter, +1 per spin | shown for each spin |

Because maczo publishes `SHA-256(serverSeed)` (the *commitment*) **before** you spin, it is
cryptographically impossible for the operator to change the server seed afterward to give you a worse
result — any change would break the hash.

### Outcome formula (identical to the server's `ProvablyFairWheel`)

```
commitment = SHA-256(serverSeed)                                      # hex, lowercase
digest     = HMAC-SHA256(key = serverSeed, msg = "clientSeed:nonce:cursor")   # cursor starts at 0
```

The 32-byte digest is read as four **8-byte big-endian unsigned integers**. To keep the odds exactly as
configured (no "modulo bias"), we use **rejection sampling**: any word `>= (2^64 - 2^64 mod totalWeight)`
is skipped and the next word is used (if all four are skipped, `cursor` increases and we re-HMAC —
astronomically rare). The first accepted word is reduced `mod totalWeight` and walked across the
cumulative segment weights to land on one segment.

## How to verify your own spins

1. On the Lucky Spin page, note the **Server Seed Hash** (commitment) shown *before* spinning, your
   **Client Seed**, and the **Nonce** for each spin.
2. When you're ready, press **Rotate** — the site permanently retires the seed and reveals the raw
   **Server Seed**.
3. Confirm `SHA-256(serverSeed)` equals the commitment you saw earlier (proves the seed was fixed
   in advance).
4. Reproduce each spin's segment from `(serverSeed, clientSeed, nonce, weights)`.

### Browser (offline)
Live at **https://maczo-co.github.io/lucky-spin/** — or open [`index.html`](./index.html) locally (double-click,
no server, no internet). Paste your values, press **Verify spin**. View source to confirm nothing is sent anywhere.

The Lucky Spin page links here with your spin data pre-filled so it computes instantly, e.g.
`https://maczo-co.github.io/lucky-spin/?server=…&commit=…&client=…&nonce=0&w=<segments>`.

### Node.js (zero dependencies)
```bash
node verify.js --commit 8ecf55c9... --server 4583aafe... \
  --client my_seed --nonce 0 \
  --weights '[{"value":0.1,"weight":300},{"value":0.2,"weight":250}]'

# run the bundled test vectors
node verify.js --selftest
```

### Test vectors
[`test-vectors.json`](./test-vectors.json) contains known `(serverSeed, clientSeed, nonce, weights) →
expectedIndex` cases generated by an **independent** reference implementation. `node verify.js --selftest`
must reproduce every one — if it does, the code here matches the specification.

## About the `weights` (segment odds)

`weights` are the relative frequencies of each wheel segment (the paytable). The reproduced
`segmentIndex` only means something if the weights you feed the verifier are the real ones. maczo
publishes the segment weights in full — the current Lucky Spin table is in
[`weights.maczo.json`](./weights.maczo.json), and every **Verify** link from the Lucky Spin page on
[maczo.co](https://maczo.co) already embeds the exact weights used for that spin, so you never have to
enter them by hand. The [Provably Fair help page](https://maczo.co/help) also documents the algorithm.

> The prize **values** are printed on the wheel; publishing the **weights** here is what makes the
> outcome fully reproducible — exactly like Stake/BC.Game publishing their game math.
